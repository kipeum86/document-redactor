# document-redactor v1.1.0

_Released 2026-04-13_

## What this is

A one-file, offline, in-browser DOCX redactor for lawyers who want to hand their documents to ChatGPT / Claude / any other LLM — but can't, because the file has company names, counterparty details, phone numbers, addresses, 주민등록번호, account numbers, or case numbers that must not leave the machine.

v1.1 makes the review surface dramatically better. You no longer scan a flat list in a narrow side panel — you read the contract inline, with every candidate highlighted in context, and click the highlights themselves to toggle.

## What's new in v1.1.0

### Inline document preview (the big one)

The center pane now renders the contract body as plain-text paragraphs grouped by scope (본문, 각주, 머리글, 바닥글), with every candidate wrapped in a yellow `<mark>` highlight. Click a highlight to toggle its inclusion. Tab / Enter / Space all work. Scope headers keep you oriented when reviewing footers and footnotes. The "No click-to-select in the preview" limitation from v1.0.0 is gone.

### Category-grouped review panel with add/remove UX

The right panel is now organized into 8 category sections in this order:

1. **당사자** — entity literals (checked)
2. **정의된 대리어** — defined-term labels (D9, unchecked)
3. **식별번호 (PII)** — RRN, BRN, EIN, phones (now including landlines), emails, accounts, cards (checked)
4. **금액** — won / USD / foreign currency / percentages / fractions / labeled amounts (checked)
5. **날짜 / 기간** — Korean / ISO / English dates + durations + labeled dates (checked)
6. **법인 / 인물** — corp suffixes / exec titles / honorifics / legal forms + label-driven addresses + label-driven phones (checked)
7. **법원 / 사건** — case numbers / court names / statutes / citations (checked)
8. **추측** — low-confidence heuristics (unchecked, amber background)

Plus a persistent **기타 (그 외)** catch-all section at the bottom with an always-open input — for anything the rules missed. Manual entries persist across re-analysis.

Each row has a **↓ jump** button that scrolls the document to that candidate and pulses the highlight.

### Three-state verification with a recovery path

v1.0 had "green banner or red banner." v1.1 splits the red case:

- **downloadReady (green)** — verify clean + sanity clean → download.
- **downloadWarning (amber)** — verify clean + sanity exceeded 30% drop threshold → override allowed with an explicit "경고를 이해하고 다운로드".
- **verifyFail (red)** — sensitive string survived in output → download blocked, but each survived row gets an "이 항목 검토" button that jumps you back to review with the offender focused. No more "Start over and lose all my selections."

### Field & hyperlink leak vectors closed

A real leak surfaced during QA on a Pearl Abyss contract: `contact@pearlabyss.com` was detected and redacted in the visible display, but the same address survived inside a `<w:instrText>HYPERLINK "mailto:...">` field instruction. The round-trip verifier caught it, but the UI couldn't explain why.

v1.1 adds a three-layer fix:

- `flatten-fields.ts` pre-pass strips all `<w:fldChar>` / `<w:instrText>` / `<w:fldSimple>` field machinery and unwraps `<w:hyperlink>` before detection runs, so nothing reaches the redactor inside a field wrapper.
- `redactInstrText` safety-net scrubs any surviving field instruction content as a second pass, in case exotic Word exports get past the flatten regex.
- `verify.ts` now also scans every `word/_rels/*.rels` file for surviving sensitive strings in `Target` attributes. Orphaned hyperlink rels entries are flagged if they still carry URL-form sensitive data.

### Broader PII coverage

- **Korean landlines** (02-, 031–069, 070, 050, 060, 080) are now detected as `identifiers.phone-kr-landline`.
- **Label-driven address capture** — `주소: 서울특별시 강남구 논현로 568` / `Address: 12345 Main St` now highlights the value after the label.
- **Label-driven phone capture** — `전화번호: 02-3446-3727` / `Phone Number: +82-2-3446-3727` captures even unusual phone formats when a label prefix is present.

Total detection rule count: **49 regex rules + 5 structural parsers + 4 heuristics** (vs 8 regex rules in v1.0). Phase 0 characterization tests still pass byte-for-byte.

### UI polish

- Seed editor removed from the sidebar — the Phase 1 detection rules + Phase 2 per-category manual-add covers what it was for. Sidebar is now just the level selector + "What Standard covers" checklist.
- `toggleSelection` now reliably triggers Svelte reactivity (defensive Set reassignment).
- "Back to review" escape hatch from verifyFail preserves all selections.
- "+ 추가" per-category input + always-open input for 기타 catch-all.

Save both files to the same directory and run:

```bash
sha256sum -c document-redactor.html.sha256
# expected output:
#   document-redactor.html: OK
```

If you see `OK`, the file is byte-identical to what the author published. Anything else — do not run it.

## Engineering notes

- **1,712 automated tests** covering detection, DOCX rewriting, verification, UI state machine, and ship gates (up from ~559 in v1.0).
- **Build determinism preserved** — same input + same selections → identical SHA-256.
- **Four-layer "no network" enforcement unchanged** — ESLint ban at source, build-time token scan, runtime `connect-src 'none'` CSP, SHA-256 sidecar.
- **Phase 0 characterization tests (T1–T24) pass byte-for-byte** — v1.0 detection behavior preserved exactly for the 8 original PII rules. All v1.1 additions are additive.

## Still on the roadmap

- **Paranoid-level hygiene** — embedded OLE / Excel scrub, image EXIF, revision IDs (`w:rsidR`), hidden text (`<w:vanish>`), `<w:sdt>` content controls. These are separate leak vectors that the current pipeline does not touch; they will ship in a later release.
- **Full rels scrub** — v1.1 flags surviving URLs in rels files but doesn't actively rewrite them. A future release will remove or sanitize orphaned rels entries.
- **Conservative / Paranoid tier selectors** — the sidebar dropdown is still disabled; Standard is the only implemented tier.
- **OCR / embedded image text** — still not on the roadmap. Browser-based OCR would break the single-file distribution model.
- **PDF input** — convert to DOCX first (Word, Google Docs, or your PDF tool's export).

## Upgrading from v1.0.0

Just replace the HTML file. No installation, no migration, no state to carry over.

If you had workflows built around the v1.0 sidebar seed editor: the seed list is no longer user-editable from the UI. The Phase 1 structural parsers and the catch-all 기타 section cover its role.
